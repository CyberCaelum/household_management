### 1. 权限控制漏洞 - 结束会话接口

**文件**: `CustomerServiceController.java` (第125-155行)

**问题描述**:
用户A可以通过构造请求结束用户B的会话，只需在请求体中传入 `userId=B的ID`。当前代码没有验证当前用户是否有权结束指定会话。

```java
// TODO 标注已明确指出问题
// TODO 没有判断权限，当前用户是否有权结束指定会话
@PostMapping("/session/end")
public Result endSession(@RequestBody SessionEndDTO sessionEndDTO) {
    Long currentUserId = BaseContext.getUserId();
    
    // 如果用户没有提供userId，使用当前用户ID
    if (sessionEndDTO.getUserId() == null) {
        sessionEndDTO.setUserId(currentUserId);
    }
    // ... 直接调用 service 结束会话，没有权限验证
}
```

**修复建议**:
```java
@PostMapping("/session/end")
public Result endSession(@RequestBody SessionEndDTO sessionEndDTO) {
    Long currentUserId = BaseContext.getUserId();
    Integer currentRole = BaseContext.getRole();
    
    // 如果用户没有提供userId，使用当前用户ID
    if (sessionEndDTO.getUserId() == null) {
        sessionEndDTO.setUserId(currentUserId);
    }
    
    // 权限验证：只能结束自己的会话，或者客服/管理员可以结束与其相关的会话
    if (!currentUserId.equals(sessionEndDTO.getUserId()) 
            && currentRole != RoleConstant.ADMIN 
            && currentRole != RoleConstant.CUSTOMER_SERVICE) {
        return Result.error("无权结束该会话");
    }
    
    // 如果是客服，验证该会话是否属于此客服
    if (currentRole == RoleConstant.CUSTOMER_SERVICE) {
        Map<String, String> session = customerServiceService.getUserSession(sessionEndDTO.getUserId());
        if (session == null || !String.valueOf(currentUserId).equals(session.get("csId"))) {
            return Result.error("无权结束该会话");
        }
    }
    // ...
}
```

### 2. findBestCustomerService 负载计算逻辑错误

**文件**: `CustomerServiceServiceImpl.java` (第299-339行)

**问题描述**:
`minLoad` 和 `maxSessions` 的初始值设置为 `Integer.MAX_VALUE`，导致第一次比较时的 `minLoadRate` 计算逻辑错误。

```java
private Long findBestCustomerService(){
    // ...
    int minLoad = Integer.MAX_VALUE;      // 初始为最大整数
    int maxSessions = Integer.MAX_VALUE;  // 初始为最大整数
    
    for (String csIdStr : onlineCsIds){
        // ...
        double loadRate = (double) currentSessions / csMaxSessions;
        // 第一次比较时：MAX_VALUE / MAX_VALUE = 1.0
        double minLoadRate = (double) minLoad / maxSessions;
        
        // 第一次比较时，任何 loadRate < 1.0 的都会选中
        if (loadRate < minLoadRate) {
            minLoad = currentSessions;
            maxSessions = csMaxSessions;
            bestCsId = csId;
        }
    }
    return bestCsId;
}
```

**问题分析**:
- `minLoad` 和 `maxSessions` 分别来自不同的客服（初始值不代表任何客服）
- 这种计算方式在数学上是不正确的，可能导致非最优的客服被选中

**修复建议**:
```java
private Long findBestCustomerService(){
    Set<String> onlineCsIds = stringRedisTemplate.opsForSet().members(
            CustomerServiceRedisKeyConstant.CS_ONLINE_ALL_KEY
    );
    if (CollectionUtils.isEmpty(onlineCsIds)){
        return null;
    }
    
    Long bestCsId = null;
    double minLoadRate = Double.MAX_VALUE;
    
    for (String csIdStr : onlineCsIds){
        Long csId = Long.valueOf(csIdStr);
        String onlineKey = CustomerServiceRedisKeyConstant.getCsOnlineKey(csId);
        Map<Object, Object> entries = stringRedisTemplate.opsForHash().entries(onlineKey);

        if (CollectionUtils.isEmpty(entries)) {
            continue;
        }
        
        int currentSessions = Integer.parseInt((String) entries.getOrDefault("currentSessions", "0"));
        int csMaxSessions = Integer.parseInt((String) entries.getOrDefault("maxSessions", "10"));
        
        if (csMaxSessions <= 0 || currentSessions >= csMaxSessions) {
            continue;
        }
        
        double loadRate = (double) currentSessions / csMaxSessions;
        if (loadRate < minLoadRate) {
            minLoadRate = loadRate;
            bestCsId = csId;
        }
    }
    return bestCsId;
}
```

### 5. 会话已存在时未验证客服是否在线

**文件**: `CustomerServiceServiceImpl.java` (第199-209行)

**问题描述**:
当用户已有会话时，直接返回会话中的客服ID，但没有验证该客服是否仍然在线。如果客服已经离线，用户会被分配给一个不在线的客服。

```java
if (stringRedisTemplate.hasKey(csSessionKey)){
    // 刷新会话信息...
    return CsGroupAssignmentResult.builder()
            .status(CsGroupAssignmentResult.Status.SESSION_EXISTS)
            .csId(String.valueOf(stringRedisTemplate.opsForHash().get(csSessionKey,"csId")))
            .message("用户已有会话")
            .build();
}
```

**修复建议**:
在返回已存在会话前，验证客服是否仍然在线，如果客服离线应重新分配或加入等待队列。

### 6. Feign 调用异常未正确处理

**文件**: `CustomerServiceServiceImpl.java` (第247-291行)

**问题描述**:
`joinCsToGroup` 和 `kickCsFromGroup` 方法在 Feign 调用失败时只记录了日志，没有进行任何回滚或补偿操作。

```java
private void joinCsToGroup(Long userId, String csId) {
    try {
        // ...
        openimFeignClient.joinGroup(...);
    } catch (Exception e) {
        log.error("客服加入群聊失败，userId={}, csId={}", userId, csId, e);
        // 只记录日志，会话已在Redis中创建，但客服实际上不在群聊中
    }
}
```

**影响**: 用户发送消息客服收不到，或者会话结束后客服仍在群聊中。

**修复建议**:
1. 加入群聊失败后应回滚会话创建操作
2. 或者实现重试机制
3. 或者记录失败状态并定时补偿

### 7. Token 过期时间计算可能为负数

**文件**: `OpenImServiceImpl.java` (第66-68行)

**问题描述**:
如果 OpenIM 返回的 `expireTimeSeconds < 600`，`adminTokenExpireTime` 会是过去的时间，导致每次请求都重新获取token。

```java
long expireTimeSeconds = jsonNode.get("data").get("expireTimeSeconds").asLong();
// 提前 10 分钟过期，避免边界问题
adminTokenExpireTime = System.currentTimeMillis() + (expireTimeSeconds - 600) * 1000;
```

**修复建议**:
```java
long expireTimeSeconds = jsonNode.get("data").get("expireTimeSeconds").asLong();
// 提前 10 分钟过期，但确保至少保留 60 秒
long bufferSeconds = Math.min(600, expireTimeSeconds - 60);
adminTokenExpireTime = System.currentTimeMillis() + (expireTimeSeconds - bufferSeconds) * 1000;
```

### 8. 等待队列 Set 和 List 可能不一致

**文件**: `CustomerServiceServiceImpl.java` (第347-381行)

**问题描述**:
`addToWaitingQueue` 方法先添加到 Set，再添加到 List。虽然代码在 List 添加失败时会回滚 Set，但如果 Redis 连接在两次操作之间中断，仍可能导致数据不一致。

**建议**:

2. 或定期同步检查和修复队列一致性
3. 或添加定时任务清理孤儿数据

### 9. processWaitingQueue 的无限循环风险

**文件**: `CustomerServiceServiceImpl.java` (第447-493行)

**问题描述**:
如果队列中有大量用户已有会话（SESSION_EXISTS 状态），`processWaitingQueue` 会一直循环处理直到队列为空。这在极端情况下可能导致长时间阻塞。

```java
private void processWaitingQueue() {
    while (true) {
        // ...
        // 如果用户已有会话，不需要特殊处理，直接处理下一个
        // 极端情况下可能长时间运行
    }
}
```

**建议**:
1. 添加最大处理数量限制
2. 或添加超时机制
3. 或使用分页方式处理
